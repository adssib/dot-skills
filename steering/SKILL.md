# Steering

This skill defines how to work with this developer across any project. It generates two files that configure the collaboration:

- **CLAUDE.md** — How Claude works with the developer (rules of engagement, workflow, code style)
- **PROJECT.md** — What we're building (architecture, tech stack, structure, build order)

## When This Skill Triggers

1. **New project setup** — User describes a project idea and wants to get started
2. **Explicit request** — User asks for a CLAUDE.md or PROJECT.md
3. **Session calibration** — User wants to adjust how Claude collaborates

## Workflow

### Step 1: Understand the Project

Before generating anything, gather these details from the user (ask if not provided):

- **What are we building?** (one sentence)
- **What's the tech stack?** (language, framework, key libraries)
- **Is this a learning project or a ship-it project?** (changes the tone of CLAUDE.md significantly)
- **What's the deployment target?** (local only, free tier hosting, cloud, etc.)
- **Any inspiration projects?** (repos, products, or ideas they're riffing on)

Don't interrogate — if the user gives a casual description, infer what you can and confirm assumptions.

### Step 2: Generate PROJECT.md

Read `references/project-md.md` for the template and rules. Generate a PROJECT.md tailored to the specific project. Key sections:

- Overview with one-line description
- Architecture diagram (ASCII art showing the major components and data flow)
- Tech stack table
- Project structure (directory tree with comments explaining each file)
- Build order broken into phases with milestones
- Any API/integration details
- Environment variables
- Learning objectives (if it's a learning project)

### Step 3: Generate CLAUDE.md

Read `references/claude-md.md` for the template and rules. Generate a CLAUDE.md tailored to the project. This file defines:

- Identity and core philosophy
- Rules of engagement (concepts before code, step-by-step chunks, trade-offs, etc.)
- What the developer writes vs what Claude writes
- Workflow (the step-by-step process for implementing anything)
- Commands (dev, test, build, deploy)
- Code style for the specific tech stack
- Architecture guardrails (non-negotiable rules for the project)

### Step 4: Present and Iterate

Show both files to the developer. Ask if anything needs adjusting — tone, chunk size, which parts they want to write vs have Claude write, missing sections, etc.

## Core Principles

These carry across ALL projects generated by this skill:

### Learning-First (when applicable)
- Concepts before code — explain the WHY before writing the HOW
- Step-by-step chunks with review gates — never implement an entire feature in one shot
- Explain trade-offs for every technical decision
- Suggest learning detours when relevant concepts come up
- Explain errors thoroughly, not just fix them

### Developer Writes What Teaches
- The developer implements anything with real learning value
- Claude implements mechanical/boilerplate code autonomously
- When giving the developer something to implement: provide function signatures, docstrings, and TODO comments — NOT the implementation
- If stuck, developer asks and Claude implements WITH explanation

### Code Quality
- Type hints on all function signatures (Python) or strict TypeScript (TS)
- Comments explain WHY, never WHAT
- No magic — explain decorators, metaclasses, or unfamiliar patterns before using them
- Git discipline — each chunk = one descriptive commit

### Communication
- Ask "Want me to explain the concept first, or are you comfortable with it?"
- Break implementation plans into numbered chunks before starting
- Stop after each chunk and wait for review
- Question unclear requirements — don't assume
